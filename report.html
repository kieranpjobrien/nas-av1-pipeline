# NASCleanup

Personal AV1 re-encoding pipeline for a home NAS. Converts H.264/HEVC
media to AV1 using NVENC hardware encoding (RTX 4000+ GPU).

## Essentials
- Package manager: **uv** (not Poetry). Run with `uv run python -m <module>`
- `paths.py` is the single source of truth for all env-var-backed paths
- Pipeline (`pipeline/`) uses stdlib only — no pip dependencies
- Server (`server/`) uses FastAPI + uvicorn; frontend is Vite + React in `frontend/`
- Three quality profiles (protected/baseline/lossy) assigned via `control/profiles.json`

## Key Paths
- `F:\AV1_Staging\` — Local staging (state, control files, temp encodes)
- `\\KieranNAS\Media\Movies\`, `\\KieranNAS\Media\Series\` — NAS libraries
- `control_templates/` — Template JSON files for pipeline control

2. Add Skills (Level 3)

Skills are markdown protocol files that teach Claude a step-by-step procedure for a specific task. They load on demand (zero tokens when inactive) and replace the need to re-explain workflows each session.

Citadel puts these in .claude/skills/. Each skill is a directory with a SKILL.md file. For this project, three skills would cover the most common workflows:

Skill 1: /new-tool

high valuerepeating pattern

You've added 7+ tools to tools/. Each follows the same pattern: argparse CLI, __main__ entry point, paths from paths.py, entry point in pyproject.toml. A skill protocol would codify this:

  1. Create module in tools/ with standard argparse skeleton
  2. Import paths from paths.py; add new path constants if needed
  3. Add entry point to pyproject.toml
  4. Add run command to CLAUDE.md (if kept) or just confirm it works
  5. Type-check with mypy, lint with ruff

Skill 2: /pipeline-debug

medium valuecomplex workflow

Debugging the encoding pipeline involves checking state JSON, control files, ffprobe output, and logs. A skill would standardise the investigation:

  1. Read F:\AV1_Staging\pipeline_state.json for current status
  2. Check control files (pause, skip, gentle, profiles, reencode)
  3. Identify the file being processed and its encoding parameters
  4. Check for common failure modes (disk space, NAS connectivity, GPU errors)
  5. Suggest fix or resume command

Skill 3: /add-plex-rule

medium valuedomain-specific

Adding metadata rules to plex_metadata.py follows a pattern: define match criteria, specify the action (collection, genre, content rating, label), add to the rules list. A skill captures this domain knowledge.

Citadel features to skip for skills

Citadel's skill system includes a router (/do), a chaining orchestrator (/marshal), and a registry in harness.json. These add complexity that isn't justified for 2–3 skills. Just put the skill files in .claude/skills/ and invoke them directly.

3. Add a Post-Edit Hook (Level 4)

One high-value hook: run ruff and mypy on every edited Python file. Catches issues immediately instead of at commit time. This is Citadel's post-edit.js adapted for your stack.

Proposed .claude/settings.json

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "bash -c 'FILE=\"$TOOL_INPUT_file_path\"; if [[ \"$FILE\" == *.py ]]; then uv run ruff check --fix \"$FILE\" && uv run ruff format \"$FILE\" && uv run mypy \"$FILE\" --ignore-missing-imports; fi'",
        "description": "Lint and typecheck edited Python files"
      }
    ]
  }
}

This gives you automatic quality enforcement without needing to remember to run checks. If it gets noisy, you can dial it back to ruff-only (drop mypy).

4. What NOT to Adopt from Citadel

Skip entirely

  • Campaign system (/archon, persistent campaign files, phase tracking) — designed for multi-day, multi-session features. Your tasks are typically single-session.
  • Fleet parallelism (worktree isolation, discovery relay, scope claims) — for teams or massive refactors across hundreds of files.
  • Coordination layer (instance registration, scope claiming) — multi-developer infrastructure. You're solo.
  • /do router — the 4-tier intent classifier is overhead when you have 2–3 skills. Just invoke them directly.
  • harness.json — generated config for the Citadel framework itself. Not needed without the full framework.
  • Telemetry — JSONL logging of hook timing and agent runs. Useful at institutional scale, noise for a personal project.
  • .planning/ directory tree — templates, intake, postmortems. These serve multi-session campaign state. Your project tracks state in pipeline_state.json already.
